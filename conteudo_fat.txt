ENUNCIADO - VALIDAÇÕES DE ENTRADA E TRATAMENTO UNIFORME DE ERROS NO PROJETO DELIVERY
Condições de conclusão
🎯 CONTEXTUALIZAÇÃO

Em sistemas de delivery como iFood, Uber Eats e Rappi, a robustez da aplicação é fundamental para garantir uma experiência confiável aos usuários. Imagine um cenário onde um cliente tenta fazer um pedido com dados inválidos ou um restaurante cadastra um produto sem informações obrigatórias - o sistema precisa responder de forma clara e consistente.

Na vida real, quando você preenche um formulário online e esquece de informar o CEP, o sistema não deve simplesmente "quebrar" ou retornar uma mensagem técnica incompreensível. Em vez disso, deve apresentar uma mensagem clara: "CEP é obrigatório para calcular a taxa de entrega".

Nesta aula, você implementará um sistema robusto de validações e tratamento de erros que:
• Valida dados de entrada antes de processá-los
• Retorna mensagens de erro padronizadas e compreensíveis
• Mantém a aplicação estável mesmo com dados inválidos
• Facilita a integração com front-ends e aplicações móveis

PROBLEMA REAL A SER RESOLVIDO:
Atualmente, nossa API de delivery não possui validações adequadas. Um cliente pode tentar cadastrar um pedido sem informar produtos, um restaurante pode ser criado sem nome, e quando erros ocorrem, as mensagens são técnicas e inconsistentes. Precisamos implementar um sistema que garanta a integridade dos dados e forneça feedback claro aos usuários.


🎯 OBJETIVOS DE APRENDIZAGEM

Ao final desta aula, você será capaz de:

1. APLICAR VALIDAÇÕES DE ENTRADA
   • Implementar @Valid e @Validated nos DTOs
   • Utilizar anotações de validação (@NotNull, @Size, @Min, @Max, @Email, etc.)
   • Criar validações customizadas para regras de negócio específicas

2. IMPLEMENTAR TRATAMENTO GLOBAL DE EXCEÇÕES
   • Desenvolver @ControllerAdvice com handlers de exceção
   • Tratar MethodArgumentNotValidException e EntityNotFoundException
   • Criar hierarquia de exceções customizadas

3. PADRONIZAR RESPOSTAS DE ERRO
   • Retornar payloads padronizados de erro (RFC 7807)
   • Implementar códigos HTTP apropriados para cada tipo de erro
   • Criar estrutura consistente de mensagens de erro


🌟 EXEMPLO DA VIDA REAL

CENÁRIO: VALIDAÇÃO EM PLATAFORMA DE DELIVERY

Imagine que você está desenvolvendo o sistema do iFood. Quando um restaurante tenta cadastrar um novo produto, várias validações devem ocorrer:

1. DADOS OBRIGATÓRIOS:
   • Nome do produto não pode estar vazio
   • Preço deve ser maior que zero
   • Categoria deve ser válida
   • Descrição deve ter pelo menos 10 caracteres

2. REGRAS DE NEGÓCIO:
   • Não pode existir dois produtos com o mesmo nome no mesmo restaurante
   • Preço não pode exceder R$ 500,00
   • Imagem deve ter formato válido (JPG, PNG)

3. RESPOSTA PADRONIZADA:
   Quando algo dá errado, o sistema retorna:
   ```json
   {
     "timestamp": "2024-06-04T14:30:00",
     "status": 400,
     "error": "Dados inválidos",
     "message": "Erro de validação nos dados enviados",
     "path": "/api/produtos",
     "details": {
       "nome": "Nome é obrigatório",
       "preco": "Preço deve ser maior que zero",
       "categoria": "Categoria inválida"
     }
   }
   ```

BENEFÍCIOS:
• Front-end sabe exatamente quais campos corrigir
• Usuário recebe feedback claro e acionável
• Sistema mantém integridade dos dados
• Desenvolvedores podem debugar facilmente


📋 ATIVIDADES PRÁTICAS
ATIVIDADE 1: IMPLEMENTAÇÃO DE VALIDAÇÕES DE ENTRADA

OBJETIVO: Implementar validações robustas nos DTOs do projeto delivery

TAREFAS:

1.1 CRIAR DTOs COM VALIDAÇÕES
• Implementar RestauranteDTO com validações:
  - Nome: obrigatório, entre 2 e 100 caracteres
  - Categoria: obrigatória, valores permitidos
  - Telefone: formato válido (10-11 dígitos)
  - Taxa de entrega: valor positivo
  - Tempo de entrega: entre 10 e 120 minutos

• Implementar ProdutoDTO com validações:
  - Nome: obrigatório, entre 2 e 50 caracteres
  - Preço: obrigatório, maior que zero, máximo R$ 500
  - Categoria: obrigatória
  - Descrição: mínimo 10 caracteres

• Implementar PedidoDTO com validações:
  - Cliente ID: obrigatório
  - Restaurante ID: obrigatório
  - Itens: lista não vazia
  - Endereço de entrega: obrigatório

1.2 VALIDAÇÕES CUSTOMIZADAS
• Criar validação @ValidCEP para formato de CEP
• Implementar @ValidTelefone para números brasileiros
• Desenvolver @ValidHorarioFuncionamento para formato HH:MM-HH:MM
• Criar @ValidCategoria para categorias permitidas

ENTREGÁVEIS:
• Classes DTO com todas as anotações de validação
• Validadores customizados funcionais
• Testes unitários das validações


ATIVIDADE 2: TRATAMENTO GLOBAL DE EXCEÇÕES

OBJETIVO: Implementar sistema robusto de tratamento de erros

TAREFAS:

2.1 HIERARQUIA DE EXCEÇÕES
• Criar BusinessException (exceção base para regras de negócio)
• Implementar EntityNotFoundException (entidade não encontrada)
• Desenvolver ValidationException (erros de validação)
• Criar ConflictException (conflitos de dados)

2.2 GLOBAL EXCEPTION HANDLER
• Implementar @ControllerAdvice GlobalExceptionHandler
• Tratar MethodArgumentNotValidException (validações @Valid)
• Tratar EntityNotFoundException (retornar 404)
• Tratar BusinessException (retornar 400)
• Tratar ConflictException (retornar 409)
• Tratar Exception genérica (retornar 500)

ENTREGÁVEIS:
• Classes de exceção customizadas
• GlobalExceptionHandler completo
• Testes de integração dos handlers


ATIVIDADE 3: PADRONIZAÇÃO DE RESPOSTAS DE ERRO

OBJETIVO: Criar estrutura padronizada para respostas de erro (RFC 7807)

TAREFAS:

3.1 ESTRUTURA DE ERRO PADRONIZADA
• Implementar ErrorResponse seguindo RFC 7807:
  - timestamp: data/hora do erro
  - status: código HTTP
  - error: tipo do erro
  - message: mensagem principal
  - path: endpoint que gerou o erro
  - details: detalhes específicos (para validações)

3.2 CÓDIGOS HTTP APROPRIADOS
• 400 Bad Request: dados inválidos
• 404 Not Found: entidade não encontrada
• 409 Conflict: conflito de dados
• 422 Unprocessable Entity: regra de negócio violada
• 500 Internal Server Error: erro interno

ENTREGÁVEIS:
• Classe ErrorResponse padronizada
• Mapeamento correto de códigos HTTP
• Documentação dos tipos de erro


ATIVIDADE 4: TESTES E VALIDAÇÃO DO SISTEMA

OBJETIVO: Validar o funcionamento completo do sistema de validações

TAREFAS:

4.1 TESTES DE CENÁRIOS DE ERRO
• Testar cadastro de restaurante com dados inválidos
• Validar criação de produto com preço negativo
• Testar pedido sem itens
• Verificar busca de entidade inexistente
• Validar conflito de dados únicos

CENÁRIOS DE TESTE OBRIGATÓRIOS:
1. POST /api/restaurantes com nome vazio → 400 Bad Request
2. POST /api/produtos com preço -10 → 400 Bad Request
3. GET /api/restaurantes/999 → 404 Not Found
4. POST /api/pedidos sem itens → 400 Bad Request
5. POST /api/restaurantes com telefone inválido → 400 Bad Request

ENTREGÁVEIS:
• Collection Postman com todos os cenários de teste
• Evidências de respostas padronizadas
• Relatório de testes executados

🔐 CONTEXTUALIZAÇÃO

Em plataformas de delivery como iFood, Uber Eats e Rappi, a segurança é fundamental para proteger dados sensíveis dos usuários, restaurantes e entregadores. Imagine um cenário onde qualquer pessoa pode acessar informações de pedidos, dados pessoais de clientes ou alterar preços de produtos sem autorização - isso seria um caos total!

Na vida real, quando você faz login no aplicativo do iFood, o sistema precisa:
• Verificar se você é realmente quem diz ser (autenticação)
• Determinar o que você pode ou não fazer (autorização)
• Manter sua sessão segura durante toda a navegação
• Proteger suas informações pessoais e de pagamento

PROBLEMA REAL A SER RESOLVIDO:
Atualmente, nossa API de delivery está completamente aberta - qualquer pessoa pode criar, visualizar, editar ou deletar restaurantes, produtos e pedidos. Isso representa um risco enorme de segurança. Precisamos implementar um sistema robusto de autenticação e autorização que:

1. CONTROLE DE ACESSO POR PERFIL:
   • Clientes: podem apenas fazer pedidos e ver seu histórico
   • Restaurantes: podem gerenciar seus produtos e ver pedidos recebidos
   • Administradores: têm acesso total ao sistema
   • Entregadores: podem ver pedidos para entrega

2. PROTEÇÃO DE DADOS SENSÍVEIS:
   • Informações pessoais dos usuários
   • Dados de pagamento e transações
   • Relatórios financeiros dos restaurantes
   • Métricas do sistema

3. RASTREABILIDADE:
   • Quem fez cada ação no sistema
   • Quando cada operação foi realizada
   • Logs de segurança para auditoria

EXEMPLO PRÁTICO:
Quando um restaurante tenta acessar a lista de seus produtos, o sistema deve:
1. Verificar se o token JWT é válido
2. Extrair o ID do restaurante do token
3. Permitir acesso apenas aos produtos desse restaurante específico
4. Bloquear tentativas de acesso a produtos de outros restaurantes


🎯 OBJETIVOS DE APRENDIZAGEM

Ao final desta aula, você será capaz de:

1. CONFIGURAR SPRING SECURITY
   • Implementar configuração de segurança personalizada
   • Definir endpoints públicos e protegidos
   • Configurar filtros de autenticação
   • Gerenciar usuários e perfis de acesso

2. IMPLEMENTAR AUTENTICAÇÃO JWT
   • Gerar tokens JWT seguros
   • Validar tokens em requisições
   • Implementar refresh tokens
   • Gerenciar expiração de tokens

3. APLICAR AUTORIZAÇÃO POR PERFIS
   • Definir roles (CLIENTE, RESTAURANTE, ADMIN, ENTREGADOR)
   • Proteger endpoints por perfil de usuário
   • Implementar controle de acesso granular
   • Validar permissões em tempo de execução

4. CRIAR SISTEMA DE USUÁRIOS
   • Implementar entidade Usuario com perfis
   • Criar endpoints de registro e login
   • Gerenciar senhas com hash seguro
   • Implementar recuperação de senha


🌟 EXEMPLO DA VIDA REAL

CENÁRIO: SISTEMA DE SEGURANÇA DO iFOOD

Imagine que você está desenvolvendo o sistema de segurança do iFood. Vamos analisar como funciona na prática:

1. PROCESSO DE LOGIN:
   • Cliente abre o app e insere email/senha
   • Sistema valida credenciais no banco de dados
   • Se válidas, gera um token JWT com informações do usuário
   • Token é enviado para o app e armazenado localmente
   • Todas as próximas requisições incluem este token

2. ESTRUTURA DO TOKEN JWT:
   ```json
   {
     "sub": "cliente123@email.com",
     "userId": 456,
     "role": "CLIENTE",
     "restauranteId": null,
     "iat": 1717516800,
     "exp": 1717603200
   }
   ```

3. CONTROLE DE ACESSO POR PERFIL:

   CLIENTE (role: CLIENTE):
   • ✅ GET /api/restaurantes (listar restaurantes)
   • ✅ GET /api/produtos (ver cardápios)
   • ✅ POST /api/pedidos (fazer pedidos)
   • ✅ GET /api/pedidos/meus (ver seus pedidos)
   • ❌ POST /api/produtos (criar produtos)
   • ❌ GET /api/pedidos (ver todos os pedidos)

   RESTAURANTE (role: RESTAURANTE):
   • ✅ GET /api/produtos/meus (ver seus produtos)
   • ✅ POST /api/produtos (criar produtos)
   • ✅ GET /api/pedidos/recebidos (pedidos para seu restaurante)
   • ❌ GET /api/pedidos (ver pedidos de outros restaurantes)
   • ❌ DELETE /api/restaurantes/{id} (deletar outros restaurantes)

   ADMIN (role: ADMIN):
   • ✅ Acesso total a todos os endpoints
   • ✅ Relatórios e métricas do sistema
   • ✅ Gerenciamento de usuários

4. FLUXO DE AUTORIZAÇÃO:
   ```
   1. Cliente faz requisição: GET /api/pedidos/meus
   2. Filtro JWT extrai token do header Authorization
   3. Valida assinatura e expiração do token
   4. Extrai userId e role do token
   5. Controller verifica se role tem permissão
   6. Se autorizado, executa a operação
   7. Retorna apenas dados do próprio cliente
   ```

5. BENEFÍCIOS DO SISTEMA:
   • Segurança: dados protegidos por autenticação
   • Escalabilidade: tokens stateless (sem sessão no servidor)
   • Flexibilidade: diferentes perfis de acesso

Nesta aula, exploramos os processos de implantação (deploy) e os diferentes ambientes de um sistema, além da integração contínua e entrega contínua (CI/CD). Os principais tópicos abordados foram:

Condições de conclusão
 
🎯 CONTEXTUALIZAÇÃO

Na era do desenvolvimento ágil e DevOps, a qualidade do software é fundamental para o sucesso de qualquer aplicação.
Testes automatizados são a base para garantir que mudanças no código não introduzam regressões e que novas
funcionalidades funcionem conforme esperado.

No contexto do nosso projeto de delivery, imagine que você é o desenvolvedor responsável por uma funcionalidade
crítica: o processamento de pedidos. Cada vez que um cliente faz um pedido, várias operações acontecem:
- Validação dos dados do cliente
- Verificação de disponibilidade dos produtos
- Cálculo do valor total
- Atualização do estoque
- Persistência no banco de dados

Sem testes automatizados, como você garantiria que essas operações funcionam corretamente após cada mudança no código?


🚨 PROBLEMATIZAÇÃO - CENÁRIO REAL

SITUAÇÃO: Você trabalha em uma startup de delivery que está crescendo rapidamente. A equipe de desenvolvimento
precisa implementar novas funcionalidades semanalmente, mas recentemente alguns bugs críticos chegaram à produção:

📱 CASO 1 - Bug no Cálculo de Preços:
Um desenvolvedor alterou a lógica de desconto e, sem perceber, quebrou o cálculo de preços para pedidos com mais
de 3 itens. Resultado: clientes pagaram valores incorretos por 2 dias até o bug ser descoberto.

🏪 CASO 2 - Falha na Validação de Estoque:
Uma mudança na validação de estoque permitiu que produtos esgotados fossem vendidos. Resultado: 50 pedidos
cancelados e clientes insatisfeitos.

👥 CASO 3 - Regressão no Cadastro de Clientes:
Uma atualização na API quebrou o endpoint de cadastro de novos clientes. Resultado: nenhum cliente novo conseguiu
se registrar por 6 horas.

💡 REFLEXÃO:
- Como esses problemas poderiam ter sido evitados?
- Qual o custo de bugs em produção vs. tempo investido em testes?
- Como garantir que mudanças futuras não quebrem funcionalidades existentes?
- Como validar que novas funcionalidades funcionam corretamente antes do deploy?

🎯 SOLUÇÃO:
Implementar uma suíte robusta de testes automatizados que:
- Testa unidades individuais de código (testes unitários)
- Valida a integração entre componentes (testes de integração)
- Garante que a API funciona corretamente (testes de controller)
- Executa automaticamente a cada mudança no código

📝 OBSERVAÇÕES IMPORTANTES

⚠️ PONTOS DE ATENÇÃO:

1. ISOLAMENTO DE TESTES:
   • Cada teste deve ser independente
   • Use @DirtiesContext quando necessário
   • Limpe dados entre testes de integração

2. NOMENCLATURA:
   • Use convenção: should_ReturnExpectedResult_When_ConditionMet
   • Seja descritivo nos nomes dos métodos de teste
   • Organize testes em classes separadas por funcionalidade

3. PERFORMANCE:
   • Testes unitários devem ser rápidos (< 100ms cada)
   • Use mocks para evitar dependências externas
   • Configure banco em memória para testes de integração

4. MANUTENIBILIDADE:
   • Evite duplicação de código nos testes
   • Use métodos auxiliares para setup comum
   • Mantenha testes simples e focados

5. COBERTURA:
   • Foque na qualidade, não apenas na quantidade
   • Teste comportamentos, não implementação
   • 80% de cobertura é o mínimo, não o objetivo final

💡 DICAS DE SUCESSO:
• Escreva testes antes de implementar (TDD)
• Execute testes frequentemente durante desenvolvimento
• Use testes como documentação do comportamento esperado
• Refatore testes junto com código de produção

Aula 11 – Escalabilidade e Performance
Condições de conclusão
1. Conceitos Utilizados
Nesta aula, discutimos os conceitos de escalabilidade e performance em sistemas distribuídos, com foco em aplicações web como o sistema de delivery. Escalabilidade refere-se à capacidade do sistema de lidar com aumento de carga, seja verticalmente (melhor hardware) ou horizontalmente (mais servidores). Performance envolve otimizações para reduzir latência, aumentar throughput e melhorar a experiência do usuário.

2. Aplicação no Dia a Dia
No sistema de delivery, aplicamos técnicas como:
✅ Cache para reduzir consultas repetidas ao banco de dados
✅ Balanceamento de carga para distribuir requisições entre servidores
✅ Uso de CDN para acelerar entrega de conteúdo estático
✅ Monitoramento contínuo para identificar gargalos e falhas
✅ Otimização de consultas SQL e índices
✅ Escalabilidade automática em nuvem para ajustar recursos conforme demanda
